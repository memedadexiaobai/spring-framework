Java 类加载器（ClassLoader）采用 **“双亲委派模型”** 形成 **三层（JDK8）→ 四层（JDK9+模块系统）** 的 **严格层次树**，**自底向上检查，自顶向下加载**，保证 **核心类不会被覆盖**，也实现 **隔离与共享** 的平衡。

---

### 1  JDK 8 经典三层

```
Bootstrap ClassLoader
         ↑
Extension  ClassLoader  （aka Platform）
         ↑
Application ClassLoader （aka System）
         ↑
Custom  ClassLoader     （用户/容器自己实现）
```

| 加载器 | 负责路径 | 实现语言 | 获取方式 |
|---|---|---|---|
| **Bootstrap** | `%JAVA_HOME%/lib/rt.jar` 等 | **C++** | `null` |
| **Extension** | `%JAVA_HOME%/lib/ext/*` | Java | `ClassLoader.getPlatformClassLoader()` |
| **Application** | `-cp` / `CLASSPATH` / 应用 jar | Java | `ClassLoader.getSystemClassLoader()` |

---

### 2  JDK 9+ 模块系统 → 四层

```
Boot Layer
  ├─ Bootstrap ClassLoader  （java.base 等模块）
  ├─ Platform ClassLoader   （平台模块）
  ├─ Application ClassLoader（应用模块）
  └─ Custom / ModuleLayer   （用户模块层）
```

- **平台类加载器**不再加载 `ext`，而是加载 **平台模块**（`java.xml`, `java.sql` 等）。
- **应用类加载器**加载 **应用模块路径**（`--module-path`）。
- **模块层（ModuleLayer）** 支持 **多根加载器**，实现 **模块隔离**。

---

### 3  双亲委派流程（通用）

1. **子加载器收到加载请求** → 先 **向上委托** 给父加载器；
2. **父加载器无法加载** → 再 **由子加载器自己尝试**；
3. 最终 **保证核心类（`java.lang.*`）始终由 Bootstrap 加载**，防止恶意覆盖。

---

### 4  常见自定义/框架级加载器

| 加载器 | 目的 |
|---|---|
| **Tomcat WebAppClassLoader** | 每个 Web 应用独立 `WEB-INF/lib`，隔离冲突 |
| **Spring Boot LaunchedURLClassLoader** | 加载嵌套 JAR（`BOOT-INF/lib`） |
| **OSGi BundleClassLoader** | 按 Bundle 版本隔离，支持动态更新 |
| **JDK 9 ModuleLayer** | 模块层隔离，支持多版本模块并存 |

---

### 5  一张图速记

```
Bootstrap  ↑  C++      核心类
Platform   ↑  Java     平台模块/扩展
System     ↑  Java     应用类
Custom     ↑  Java     容器/框架/用户隔离
```

> **“自上而下加载，自下而上检查；祖宗能加载，子孙不重复；模块时代多一层，隔离共享两不误。”**
